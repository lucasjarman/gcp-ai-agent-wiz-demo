# Wiz Cross-Origin Threat Detection Scenarios

## Objective

Create small, controlled scenarios showing how Wiz combines runtime telemetry,
GKE or GCP audit logs, AI-workload context, and Security Graph relationships.
All data and identities are synthetic. The scenarios are intentionally narrow
and are not production reference patterns.

## Current Architecture

- Cloudflare and a Kubernetes NetworkPolicy restrict access to the public GKE
  application.
- The FastAPI and LangChain agent runs in the Sensor-protected `ai-dlc-demo`
  namespace on GKE Autopilot.
- GKE Workload Identity maps the application KSA to `ai-dlc-demo-sa`.
- A project log sink sends GCP Audit, GKE Audit, and selected Data Access logs
  to the Wiz Cloud Events integration.
- Tool subprocesses run inside the long-lived AI workload container, retaining
  the workload's runtime and AI context.

## Repeatable Agent Driver

Permanent scenarios use this fixed flow:

```text
Exact hidden chat prompt + operator run token
  -> server-side scenario gate
  -> deterministic fixed commands
  -> bounded execution in the existing app container
  -> sanitized tool trace and result
```

Controls:

- Require `DEMO_SCENARIOS_ENABLED=true`, empty chat history, exact prompt digest,
  and a separate operator-token digest.
- Store the prompt and token in BWS; Kubernetes receives only SHA-256 digests.
- Never expose a scenario tool to ordinary agent prompts or accept command,
  project, identity, or resource arguments from the user.
- Suppress credential output and raw cloud errors.
- Serialize runs, enforce a 30-minute cooldown, and allow at most six runs per
  UTC day.
- Preserve Cloudflare, the origin NetworkPolicy, and bounded Python execution.

## Scenario 1: Pod Token Access

Purpose: demonstrate a user entering an AI workload and accessing its mounted
Kubernetes identity.

Controlled trigger:

```bash
kubectl exec -n ai-dlc-demo deploy/ai-dlc-demo -- \
  sh -c 'cat /var/run/secrets/kubernetes.io/serviceaccount/token | wc -c'
```

Only the token length is returned. The token is not copied, logged, or used.

Expected evidence:

| Origin | Rule | Result |
| --- | --- | --- |
| `WIZ_SENSOR` | `cer-sen-id-265` | Mounted KSA token accessed by a text reader |
| `GCP_GKE_AUDIT_LOGS` | `cer-kubernetes-compute-unusualPodTokenRetrievalExec` | Pod token retrieval through pod exec |

This smoke test succeeded on 2026-07-23.

## Scenario 2: Kubernetes Discovery Session

Purpose: demonstrate multiple Kubernetes discovery actions in one interactive
execution scope.

Candidate built-in correlation:
`cer-correlation-id-321`, **Kubernetes permissions discovery and secret access
operations in a session**.

The rule is enabled in the tenant, but it requires every Sensor event to carry
the `interactiveSession` context tag and to share an execution scope. It is a
secondary scenario because the web-driven agent process is non-interactive.
Do not claim this correlation without first validating a real TTY-based path.

## Scenario 3: Agent-Driven GCP Persistence

### Purpose

Demonstrate an AI-agent tool being abused to enumerate GCP service accounts and
create a user-managed key. Wiz correlates commands observed by the Runtime
Sensor with the real GCP audit event generated by the key creation.

This is the recommended permanent scenario.

### Built-in Correlation

The tenant has enabled built-in rule `cer-correlation-id-90`:
**Enumeration and creation of private-key attempt for a service account
originated from GKE container**.

It generates a Critical Issue from these inputs:

| Origin | Rule or event | Required activity |
| --- | --- | --- |
| `WIZ_SENSOR` | `cer-sen-id-527` | `gcloud iam service-accounts list` |
| `WIZ_SENSOR` | `cer-sen-id-528` | `gcloud iam service-accounts keys create` |
| `GCP_AUDIT_LOGS` | IAM activity | `google.iam.admin.v1.CreateServiceAccountKey` |

The rule runs every ten minutes over a 20–22-minute window. Its Workload
Identity branch joins the Sensor-observed pod GSA to the normalized GCP audit
actor. It does not require DSPM sensitivity, a behavioral profile, a custom
TDR, or a custom Wiz API identity.

### Canary Architecture

Create one service account in `lucas-ai-agent-demo`:

- `ai-dlc-rule90-canary`

The identity has no project or resource roles. The application GSA receives
`roles/iam.serviceAccountKeyAdmin` only on that identity. Admin Activity logs
record the key lifecycle and the existing project sink routes them to Wiz. A
project-level override disables the inherited managed constraint that would
otherwise block service-account key creation.

No new project, billing-account attachment, bucket, custom TDR, or Wiz write
credential is required. A real user-managed key exists briefly during each
run, but its local material and cloud resource are deleted immediately.

### Fixed Execution

After the prompt and operator-token gate, the long-lived agent process launches
these direct `shell=False` child processes:

```text
gcloud iam service-accounts list \
  --project=lucas-ai-agent-demo \
  --filter=email:ai-dlc-rule90-canary \
  --limit=1 --format=value(email) --quiet

gcloud iam service-accounts keys create <temporary-path> \
  --iam-account=ai-dlc-rule90-canary@lucas-ai-agent-demo.iam.gserviceaccount.com \
  --project=lucas-ai-agent-demo --quiet

gcloud iam service-accounts keys delete <created-key-id> \
  --iam-account=ai-dlc-rule90-canary@lucas-ai-agent-demo.iam.gserviceaccount.com \
  --project=lucas-ai-agent-demo --quiet
```

The application reads only `private_key_id` from the generated JSON, removes
the local JSON, and deletes that key from IAM. A fresh Cloud SDK configuration
and temporary directory are used per run.

### Verification

1. Confirm the list, create, and delete commands complete in one app-container
   execution.
2. Confirm Sensor alerts 527 and 528 are attributed to the `ai-dlc-demo` AI
   workload.
3. Confirm the `CreateServiceAccountKey` audit event arrives with the app GSA
   as actor.
4. Wait through the ten-minute evaluation cadence and confirm rule 90 creates
   or updates one Critical Issue.
5. Inspect the Issue evidence for both `WIZ_SENSOR` and `GCP_AUDIT_LOGS` origins
   and the expected GKE workload and identity relationships.

Repeated runs may update an existing Issue rather than create a new identifier.
Do not retrigger inside the application cooldown while validating ingestion.

## Rejected Alternatives

- Correlation 89 is absent from this tenant and requires sensitive-bucket DSPM
  enrichment.
- Correlation 93 received both expected source streams in a live run but did
  not produce its final correlation, so it is not used for the permanent demo.
- Correlations 91 and 92 mutate SSH project metadata.
- A custom TDR would add tenant-specific write credentials without improving
  the built-in rule-90 story.
- A fake offensive-tool user agent would simulate branding rather than execute
  a genuine attacker behavior.
